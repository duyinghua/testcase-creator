# 通用测试用例生成器技能

## 概述

本技能自动化从需求文档生成测试用例文档的过程。它分析需求内容、提取功能模块和测试点、生成结构化的Markdown测试用例文档，并将其转换为XMind思维导图格式，以便更好地可视化和管理。

**核心特性**：
- ✅ 深度需求分析，挖掘所有测试细节
- ✅ 按完整功能流程组织测试用例
- ✅ 自动生成Markdown和XMind两种格式
- ✅ **时间戳版本管理**：每次生成都添加时间戳后缀，保留所有历史版本

**🚨 核心原则（必读）**：

1. **深入分析需求**：不要浮于表面，必须逐句分析需求文档，挖掘所有测试细节
2. **充分覆盖细节**：验证项数量不设上限，核心功能的每个细节都要有对应的验证项
3. **不要人为限制**：不要为了控制数量而合并验证项，每个可独立验证的点都应单独列出
4. **质量优先于数量**：宁可测试用例详细冗长，也不要遗漏核心功能的测试细节
5. **保留历史版本**：每次生成都添加时间戳，所有版本都被保留，便于版本管理和对比

## 何时使用此技能

在以下情况下使用此技能：
- 用户提供需求文档并请求生成测试用例
- 用户要求从产品规格说明或设计文档创建测试用例
- 用户需要将现有需求文档转换为测试用例
- 用户希望生成Markdown和XMind两种格式的测试用例文档
- 用户提到关键词如"测试用例"、"test cases"、"思维导图"、"mindmap"、"需求文档"

## 工作流程

### 步骤1：读取和分析需求文档

从提供的来源（文件路径或直接内容）读取需求文档内容。**这一步是测试用例质量的关键，必须深入分析，不能浮于表面。**

#### 1.1 文档读取

**重要提示**：
- 如果需求以URL形式提供（如飞书文档），首先使用 `fetch-doc` 技能下载文档
- 如果需求以文件路径形式提供，直接读取文件内容
- 专注于文字内容，除非图片包含关键信息，否则忽略图片

#### 1.2 深度需求分析方法

**必须对需求进行深度分析，不要只停留在表面！**

##### 1.2.1 逐句分析法

对需求文档中的每一句话进行以下分析：

1. **识别关键词**：
   - 条件词：当、如果、若、满足...时、在...情况下
   - 动作词：点击、输入、选择、提交、展示、隐藏、跳转、调用
   - 数据词：创建、读取、更新、删除、保存、加载、同步
   - 状态词：启用、禁用、显示、隐藏、选中、未选中
   - 关系词：与、和、或、且、包含、属于、关联

2. **提取测试要素**：
   ```
   需求句子："当用户已触发OSA，则在页面底部展示OSA常驻入口"
   
   分析过程：
   - 条件：用户已触发OSA
   - 动作：展示
   - 对象：OSA常驻入口
   - 位置：页面底部
   
   提取验证项：
   - 验证用户已触发OSA时入口展示
   - 验证用户未触发OSA时入口不展示
   - 验证入口展示位置在页面底部
   - 验证入口为常驻特性
   ```

##### 1.2.2 业务流程分析法

对于涉及业务流程的需求：

1. **绘制流程图**：
   - 识别流程的起点和终点
   - 识别流程中的每个步骤
   - 识别流程中的判断分支
   - 识别流程中的循环结构

2. **测试每个节点**：
   - 每个流程节点都应有对应的测试点
   - 每个判断分支都应有对应的验证项
   - 每个异常退出都应有对应的异常测试

##### 1.2.3 数据流分析法

对于涉及数据交互的需求：

1. **追踪数据流**：
   - 数据从哪里来（数据源）
   - 数据如何处理（数据处理）
   - 数据到哪里去（数据存储/展示）

2. **测试每个环节**：
   - 数据获取：验证数据获取的正确性
   - 数据处理：验证处理逻辑的正确性
   - 数据存储：验证存储的完整性
   - 数据展示：验证展示的准确性

##### 1.2.4 状态转换分析法

对于涉及状态变化的需求：

1. **识别所有状态**：
   - 初始状态
   - 中间状态
   - 终止状态

2. **测试状态转换**：
   - 每个状态转换都应有对应的验证项
   - 验证状态转换的条件
   - 验证状态转换的时机
   - 验证状态转换的结果

##### 1.2.5 用户场景分析法

从用户角度分析：

1. **识别用户角色**：
   - 不同角色可能有不同的权限和操作
   - 每个角色都应有对应的测试场景

2. **分析使用场景**：
   - 正常使用场景
   - 异常使用场景
   - 极端使用场景
   - 组合使用场景

#### 1.3 分析结果整理

将分析结果整理为以下结构：

1. **功能模块清单**：
   - 列出所有功能模块
   - 列出每个模块的子功能
   - 列出每个子功能的功能点

2. **业务规则清单**：
   - 列出所有业务规则
   - 列出所有约束条件
   - 列出所有边界条件

3. **异常场景清单**：
   - 列出所有可能的异常
   - 列出所有边界情况
   - 列出所有特殊场景

4. **测试数据清单**：
   - 列出所需的测试数据
   - 列出数据的准备方法
   - 列出数据的清理方法

### 步骤2：生成Markdown测试用例文档

**重要提示**：必须严格遵守 `references/testcase_template.md` 中定义的格式规范！

#### 2.0 文件命名规则

**核心原则**：每次生成都添加时间戳后缀，便于版本管理和历史追溯

本技能使用自动化脚本为每个测试用例文件添加时间戳后缀，确保所有版本都被保留，便于对比和回溯。

##### 2.0.1 命名规则

**文件名格式**：
```
{需求文档名称}-v{MMdd_HHmmss}.md
{需求文档名称}-v{MMdd_HHmmss}.xmind
```

**时间戳格式**：
- 格式：`MMdd_HHmmss`（月日_时分秒）
- 示例：`0318_143052` 表示3月18日14:30:52

##### 2.0.2 使用方法

**调用脚本生成文件名**：
```bash
python3 scripts/generate_filename.py "<base_name>" "<output_dir>"

# 示例
python3 scripts/generate_filename.py "专属管家产品V1.0测试用例" "/Users/xxx/downloads"

# 输出JSON结果
{
  "md_file": "/Users/xxx/downloads/专属管家产品V1.0测试用例-v0318_143052.md",
  "xmind_file": "/Users/xxx/downloads/专属管家产品V1.0测试用例-v0318_143052.xmind",
  "base_name": "专属管家产品V1.0测试用例-v0318_143052"
}
```

##### 2.0.4 优势

- ✅ **版本清晰**：每个版本都有时间戳，易于识别生成时间
- ✅ **避免覆盖**：所有版本都被保留，不会丢失历史数据
- ✅ **便于对比**：可以通过时间戳对比不同版本的测试用例
- ✅ **简化管理**：不需要检查文件是否存在，逻辑简单直接

#### 2.1 标准格式结构

按照以下标准结构创建Markdown测试用例文档：

```markdown
# {产品需求名称} - 测试用例

## 一、{功能模块名称}

### 1.1 {子功能名称}

#### 1.1.1 {功能点名称}
- **测试点：{测试点名称}**
  - [ ] {验证项1}
  - [ ] {验证项2}
  - [ ] {验证项3}

- **测试点：{测试点名称}**
  - [ ] {验证项1}
  - [ ] {验证项2}

### 1.2 {子功能名称}

#### 1.2.1 {功能点名称}
- **测试点：{测试点名称}**
  - [ ] {验证项1}
```

#### 2.2 格式规范要求

**必须遵守的格式规则**：

1. **标题层级规范**：
   - `# 一级标题`：产品名称（文档根标题）
   - `## 二级标题`：功能模块（使用"一、二、三、"等中文序号）
   - `### 三级标题`：子功能（使用"1.1、1.2、2.1、"等编号）
   - `#### 四级标题`：功能点（使用"1.1.1、1.1.2、"等编号）

2. **测试点格式规范**：
   - 必须使用粗体标记：`**测试点：XXX验证**`
   - 测试点名称应具体明确，描述验证目标
   - 示例：`**测试点：触发条件验证**`、`**测试点：展示验证**`

3. **验证项格式规范**：
   - 必须使用待办事项格式：`- [ ] 验证项内容`
   - 使用2个空格缩进表示层级关系
   - 每个验证项应具体、可执行、可验证
   - 避免模糊描述，如"验证功能正常"应改为"验证点击按钮后跳转到正确页面"

4. **文本处理规范**：
   - 保留粗体标记 `**文本**`
   - 支持emoji和特殊字符
   - 不使用checkbox标记 `[x]`，统一使用 `[ ]`

#### 2.3 测试用例生成质量标准

**生成测试用例时必须遵循以下质量标准**：

##### 2.3.1 测试覆盖完整性

每个功能模块必须覆盖以下测试类型：

1. **功能测试**（必须包含）：
   - 核心功能：主要业务流程的正确性
   - 用户交互：点击、滑动、输入等操作响应
   - 数据处理：数据的创建、读取、更新、删除
   - 业务规则：业务逻辑、约束条件、计算规则

2. **UI/UX测试**（必须包含）：
   - 界面展示：元素位置、大小、颜色、字体
   - 布局适配：不同屏幕尺寸、横竖屏切换
   - 响应性：触摸反馈、动画效果、加载状态

3. **异常场景测试**（必须包含）：
   - 网络异常：弱网、无网、超时、断网重连
   - 数据异常：空数据、错误数据、边界数据
   - 并发场景：快速点击、重复操作、多任务切换
   - 权限控制：未登录、权限不足、权限拒绝

4. **边界条件测试**（建议包含）：
   - 数据边界：最大值、最小值、临界值
   - 时间边界：超时、时区、跨天
   - 数量边界：0条、1条、最大条数

5. **性能测试**（建议包含）：
   - 加载性能：首屏加载、分页加载、懒加载
   - 响应性能：点击响应、接口响应
   - 稳定性：长时间运行、内存泄漏、电量消耗

6. **兼容性测试**（建议包含）：
   - 设备兼容：不同品牌、型号、屏幕尺寸
   - 系统兼容：不同操作系统版本
   - 版本兼容：新旧版本数据迁移、接口兼容

7. **安全测试**（建议包含）：
   - 数据安全：敏感信息加密、传输安全
   - 权限安全：越权访问、权限控制
   - 输入安全：XSS、SQL注入、参数篡改

##### 2.3.2 测试点命名规范

测试点名称应遵循以下命名模式：

1. **功能验证类**：
   - `{功能名称}验证`：如"触发条件验证"、"数据提交验证"
   - `{功能名称}功能验证`：如"登录功能验证"、"搜索功能验证"

2. **展示验证类**：
   - `{元素名称}展示验证`：如"列表展示验证"、"详情页展示验证"
   - `{状态}展示验证`：如"空状态展示验证"、"加载状态展示验证"

3. **交互验证类**：
   - `{操作名称}交互验证`：如"点击交互验证"、"滑动交互验证"
   - `{流程名称}流程验证`：如"注册流程验证"、"支付流程验证"

4. **异常验证类**：
   - `{异常类型}异常验证`：如"网络异常验证"、"数据异常验证"
   - `{边界条件}边界验证`：如"数据边界验证"、"时间边界验证"

##### 2.3.3 验证项编写规范

每个验证项必须符合SMART原则：

1. **具体性（Specific）**：
   - ❌ 错误：验证功能正常
   - ✅ 正确：验证点击"提交"按钮后，表单数据成功提交到服务器

2. **可测量性（Measurable）**：
   - ❌ 错误：页面加载快
   - ✅ 正确：页面首次加载时间不超过2秒

3. **可达成性（Achievable）**：
   - 验证项应该是在当前测试环境下可执行的
   - 不包含不可控的外部依赖

4. **相关性（Relevant）**：
   - 验证项应与测试点主题直接相关
   - 避免偏离测试目标的验证项

5. **时效性（Time-bound）**：
   - 对于性能相关的验证项，应明确时间限制
   - 示例：接口响应时间不超过500ms

##### 2.3.4 验证项数量要求

**重要原则：验证项数量不设上限，以充分覆盖功能细节为目标！**

每个测试点的验证项数量应根据以下因素灵活确定：

1. **功能复杂度**：
   - 简单功能：至少包含基本验证项
   - 中等功能：包含完整的验证场景
   - 复杂功能：详细覆盖所有可能的场景和分支

2. **功能重要性**：
   - **核心功能**：必须充分详细，不限制验证项数量，确保覆盖所有细节
   - **重要功能**：全面覆盖主要场景和异常场景
   - **次要功能**：覆盖基本功能即可

3. **需求细节程度**：
   - 需求描述详细的：应逐一转换为验证项
   - 需求描述简略的：根据业务逻辑补充验证项
   - 需求有多个条件分支的：每个分支都应有对应验证项

**核心功能测试的特别要求**：

对于核心功能，必须：
- ✅ **深入挖掘需求细节**：仔细阅读需求文档中的每一句话，不遗漏任何细节
- ✅ **覆盖所有条件分支**：if-else的每个分支都要有验证项
- ✅ **验证所有业务规则**：每个业务规则都要有对应的测试验证
- ✅ **测试所有边界情况**：最大值、最小值、临界值都要覆盖
- ✅ **覆盖异常处理**：每个可能的异常情况都要有验证项
- ✅ **验证状态转换**：如果功能涉及状态流转，每个状态转换都要测试
- ✅ **测试数据完整性**：验证数据的创建、更新、删除等操作

**验证项生成的详细指导**：

在生成验证项时，请遵循以下步骤：

1. **提取需求中的所有条件**：
   ```
   需求："当用户满足物料条件时，展示OSA入口"
   提取验证项：
   - [ ] 验证满足物料条件时入口正确展示
   - [ ] 验证不满足物料条件时入口不展示
   - [ ] 验证物料条件变化时入口动态更新
   - [ ] 验证页面刷新后入口状态保持
   - [ ] 验证物料条件为空时的处理逻辑
   ```

2. **分析需求中的所有动作**：
   ```
   需求："点击电话图标触发拨号"
   提取验证项：
   - [ ] 验证点击电话图标触发拨号功能
   - [ ] 验证拨号界面正确展示经纪人号码
   - [ ] 验证取消拨号后返回页面正常
   - [ ] 验证拨号成功后的记录
   - [ ] 验证电话权限拒绝时的提示
   - [ ] 验证无网络时的拨号提示
   ```

3. **识别需求中的所有状态**：
   ```
   需求："入口在页面底部展示"
   提取验证项：
   - [ ] 验证入口在页面底部正确展示
   - [ ] 验证入口不遮挡页面内容
   - [ ] 验证滚动页面时入口位置固定
   - [ ] 验证入口与页面底部安全距离正确
   - [ ] 验证横竖屏切换时入口位置正确
   - [ ] 验证不同屏幕尺寸下入口展示正确
   ```

4. **挖掘隐含的业务规则**：
   ```
   需求："入口物料与小区详情页共用"
   隐含验证项：
   - [ ] 验证入口物料与小区详情页一致
   - [ ] 验证物料推送策略正确执行
   - [ ] 验证物料内容符合用户画像
   - [ ] 验证物料更新逻辑正确
   - [ ] 验证物料类型完整性（工具/动态/情报/求助）
   - [ ] 验证不同业务类型展示（专属管家/交易专家/金融专家）
   ```

**不要人为限制验证项数量**：

- ❌ 错误做法：为了控制数量而合并验证项
- ✅ 正确做法：每个可独立验证的点都应该单独列出

- ❌ 错误做法：认为验证项太多会显得冗余
- ✅ 正确做法：核心功能的细节越详细越好，确保测试充分

**质量检查重点**：
- 每个验证项是否有明确的测试目标
- 验证项之间是否相互独立
- 是否有遗漏的需求细节
- 是否覆盖了所有可能的场景

#### 2.4 测试用例生成流程

生成测试用例时，请严格按照以下流程执行：

##### 2.4.1 需求分析阶段（最重要）

**目标：深入理解需求，挖掘所有测试细节**

1. **通读需求文档**：
   - 仔细阅读整个需求文档，不要跳过任何部分
   - 理解业务背景和目标用户
   - 识别关键业务流程和核心价值

2. **逐句分析需求**：
   - **对每一句话进行深度分析**，提取测试点
   - 标记所有条件判断（if/when/如果/当...时）
   - 标记所有动作操作（点击/输入/选择/提交等）
   - 标记所有状态变化（展示/隐藏/启用/禁用等）
   - 标记所有数据交互（创建/读取/更新/删除等）

3. **识别和归并功能模块**：
   
   **🚨 核心原则：按完整功能流程组织测试，而非按需求文档章节划分**
   
   **步骤1：识别章节关系**
   
   需求文档的章节通常有以下几种组织方式：
   - **按功能模块划分**：每个章节对应一个独立功能
   - **按功能维度划分**：同一功能的不同维度分散在不同章节
   - **按实现层次划分**：前端展示、后端逻辑、数据模型等分开描述
   
   **步骤2：判断章节是否需要归并**
   
   使用以下判断标准：
   
   1. **功能一致性检查**：
      - 这些章节是否描述同一个功能？
      - 用户是否会在同一个使用场景中使用？
      - 测试时是否需要一起验证？
      
   2. **引用关系检查**：
      - 章节中是否有"详见XX章节"、"见XX部分"、"XX逻辑见XX"等引用？
      - 引用的内容是否是同一功能的实现细节？
      
   3. **维度分析**：
      - 章节A描述的是功能的"是什么"（界面、交互）
      - 章节B描述的是功能的"怎么实现"（触发逻辑、业务规则）
      - 这种情况应归并为一个测试模块
   
   **步骤3：设计测试用例结构**
   
   对于每个识别出的功能模块，按以下流程组织测试：
   
   ```
   功能模块
   └── 子功能/入口/页面
       ├── 触发条件验证（什么情况下出现）
       ├── 业务规则验证（如何判断/计算）
       ├── 数据准备验证（需要什么数据）
       ├── 展示内容验证（展示什么内容）
       ├── 交互行为验证（用户如何操作）
       └── 结果处理验证（操作后的结果）
   ```
   
   **步骤4：通用示例说明**
   
   **示例1：章节描述同一功能的不同维度**
   ```
   需求文档结构：
   第一章 功能界面设计
     1.1 页面布局
     1.2 交互流程
     1.3 展示规则
   
   第三章 业务逻辑实现
     3.1 触发条件
     3.2 计算规则
     3.3 异常处理
   
   ✅ 正确的测试用例结构：
   一、XX功能
     1.1 页面/入口A
       - 触发条件验证（来自第三章）
       - 业务规则验证（来自第三章）
       - 页面展示验证（来自第一章）
       - 交互流程验证（来自第一章）
     1.2 页面/入口B
       - ...
   ```
   
   **示例2：章节描述同一功能的不同层次**
   ```
   需求文档结构：
   第二章 用户界面
     2.1 登录页面
     2.2 注册页面
   
   第四章 接口设计
     4.1 登录接口
     4.2 注册接口
   
   第六章 数据校验规则
     6.1 用户名校验
     6.2 密码校验
   
   ✅ 正确的测试用例结构：
   一、用户认证功能
     1.1 登录功能
       - 登录页面展示验证（来自第二章）
       - 登录接口调用验证（来自第四章）
       - 用户名校验规则验证（来自第六章）
       - 密码校验规则验证（来自第六章）
       - 登录成功流程验证
       - 登录失败场景验证
     1.2 注册功能
       - ...
   ```
   
   **步骤5：归并原则总结**
   
   | 需求文档章节关系 | 是否归并 | 归并方式 |
   |-----------------|---------|---------|
   | 不同功能的描述 | ❌ 不归并 | 各自独立成模块 |
   | 同一功能的不同维度 | ✅ 归并 | 融入同一模块的测试流程 |
   | 同一功能的不同层次 | ✅ 归并 | 融入同一模块的测试流程 |
   | 功能描述+实现细节 | ✅ 归并 | 实现细节融入功能测试 |
   | 功能描述+策略规则 | ✅ 归并 | 策略规则融入功能测试 |
   
   **步骤6：质量检查**
   
   归并完成后，检查以下内容：
   - [ ] 测试人员能否在一个模块下完成该功能的完整测试流程？
   - [ ] 是否需要在多个章节/模块间来回跳转？
   - [ ] 每个功能的测试是否包含：触发→处理→展示→交互→结果？
   - [ ] 是否有遗漏的需求细节？
   
   **关键要点**：
   - **测试用例是给测试人员用的**，应按测试流程组织，而非按需求文档章节组织
   - **需求文档章节**可能按产品设计思路、开发实现思路、文档阅读便利性等组织
   - **测试用例结构**应按测试执行思路、用户使用流程组织
   - 当需求文档章节与测试流程不匹配时，**测试用例应重新组织**

4. **提取业务规则**：
   - 记录所有的业务规则和约束条件
   - 每个规则都应转化为验证项
   - 注意隐含的业务逻辑

5. **挖掘异常场景**：
   - 思考每个功能可能的异常情况
   - 网络异常、数据异常、权限异常等
   - 边界条件和极限情况

6. **分析用户场景**：
   - 从用户角度思考使用场景
   - 考虑不同用户角色的行为
   - 考虑不同的使用环境

**需求分析示例**：

```
需求原文："小区在售房源列表页，进入页面满足物料条件就出，则在该页面底部展示OSA常驻入口"

深度分析：
1. 页面类型：小区在售房源列表页
2. 触发条件：满足物料条件

4. 展示位置：页面底部
5. 入口类型：OSA常驻入口

提取测试点：
- 触发条件验证（物料条件）
- 触发时机验证（进入页面）
- 展示位置验证（页面底部）
- 入口类型验证（常驻特性）
- 入口物料验证（物料类型和内容）

每个测试点进一步细化验证项...
```

##### 2.4.2 测试点设计阶段

**目标：为每个功能点设计全面的测试点**

1. **逐个功能点分析**：
   - 不要遗漏任何功能点
   - 每个功能点至少包含一个测试点
   - 复杂功能点应拆分为多个测试点

2. **设计测试点类型**：
   - 功能验证类：验证功能是否按预期工作
   - 展示验证类：验证UI展示是否正确
   - 交互验证类：验证用户交互是否流畅
   - 异常验证类：验证异常处理是否合理
   - 性能验证类：验证性能指标是否达标
   - 安全验证类：验证安全措施是否到位

3. **考虑测试场景**：
   - 正向场景：正常的业务流程
   - 反向场景：异常和错误情况
   - 边界场景：临界值和极限值
   - 并发场景：多用户或多操作

##### 2.4.3 验证项编写阶段

**目标：为每个测试点编写详细可执行的验证项**

1. **细化每个测试点**：
   - 将测试点拆解为具体的验证步骤
   - 每个验证项对应一个可执行的测试操作
   - 不限制验证项数量，确保充分覆盖

2. **使用具体描述**：
   - 避免"验证功能正常"等模糊描述
   - 使用"验证点击XX按钮后，跳转到XX页面"等具体描述
   - 包含前置条件、操作步骤、预期结果

3. **覆盖所有细节**：
   - **不要遗漏需求中的任何细节**
   - 每个细节都应有对应的验证项
   - 宁可多写，不要少写

4. **标注测试数据**：
   - 明确测试所需的数据类型
   - 标注特殊的测试数据要求
   - 说明数据准备方法

##### 2.4.4 质量检查阶段

**目标：确保测试用例的完整性和准确性**

1. **需求覆盖检查**：
   - 对照需求文档，逐条检查是否有对应测试用例
   - 确保每个需求点都被覆盖
   - 不遗漏任何功能细节

2. **格式规范检查**：
   - 检查格式是否符合 testcase_template.md 规范
   - 检查标题层级是否正确
   - 检查验证项格式是否规范

3. **验证项质量检查**：
   - 检查验证项是否具体明确
   - 检查验证项是否可执行
   - 检查验证项是否有遗漏

4. **重复性检查**：
   - 检查是否有重复的测试点
   - 检查是否有重复的验证项
   - 合并或删除重复内容

## 最佳实践

### 测试用例质量标准

遵循以下SMART原则确保测试用例质量：

1. **具体性（Specific）**：
   - 每个验证项应该具体明确
   - 避免模糊描述，如"验证功能正常"
   - 示例：验证点击"提交"按钮后，表单数据成功提交并显示成功提示

2. **独立性（Independent）**：
   - 测试用例应该独立，不依赖于执行顺序
   - 每个测试点可以单独执行
   - 验证项之间不应有依赖关系

3. **可重复性（Repeatable）**：
   - 测试用例多次执行应产生一致的结果
   - 明确前置条件和测试数据
   - 确保测试环境可重现

4. **全面性（Comprehensive）**：
   - 覆盖正向、负向和边界测试场景
   - 考虑不同的用户角色和使用场景
   - 不遗漏关键业务流程

5. **可维护性（Maintainable）**：
   - 使用清晰的命名和组织方式
   - 结构化的文档格式
   - 便于后续更新和维护

### 测试点组织原则

按照以下原则组织测试点：

1. **按模块分组**：
   - 按功能模块组织测试用例
   - 模块名称应清晰反映功能范围
   - 相关的子功能应组织在同一模块下

2. **按功能分类**：
   - 每个模块内按具体功能组织
   - 功能点应细化到可测试的粒度
   - 避免一个功能点包含过多测试内容

3. **按场景归类**：
   - 将相关的测试场景分组
   - 正常流程和异常流程分开
   - 高优先级场景优先展示

4. **按优先级标记**：
   - P0：核心功能，阻塞上线
   - P1：重要功能，影响用户体验
   - P2：次要功能，可延后测试
   - P3：优化类功能，按需测试

### 测试覆盖清单

确保覆盖以下测试场景：

**必测场景**：
- [ ] 核心业务流程（快乐路径）
- [ ] 错误处理和异常提示
- [ ] 关键数据验证
- [ ] 用户权限控制
- [ ] 网络异常场景

**重要场景**：
- [ ] 边界条件测试
- [ ] 数据完整性验证
- [ ] 接口集成验证
- [ ] 性能指标验证

**可选场景**：
- [ ] 安全性测试
- [ ] 兼容性测试
- [ ] 压力测试
- [ ] 长期运行稳定性测试

## 使用示例

### 示例1：从本地文档生成测试用例

**用户**："根据这份需求文档生成测试用例"

**助手执行步骤**：

1. **读取需求文档**
   - 读取用户提供的文档内容
   - 识别文档中的功能模块和业务逻辑

2. **分析需求**
   - 提取主要功能模块
   - 识别子功能和功能点
   - 理解业务规则和约束条件

3. **生成测试用例**
   - 按照标准格式生成Markdown文档
   - 确保覆盖功能测试、异常测试、边界测试
   - 为每个测试点编写具体可执行的验证项

4. **质量检查**
   - 检查格式是否符合testcase_template.md规范
   - 检查验证项是否具体明确
   - 检查是否有遗漏的测试场景

5. **转换为XMind**
   - 使用转换脚本生成XMind文件
   - 验证XMind文件可正常打开

6. **展示结果**
   - 展示Markdown和XMind文件
   - 提供测试统计信息
   - 给出测试建议

### 示例2：从飞书文档生成测试用例

**用户**："根据这个飞书文档 https://xxx.feishu.cn/xxx 生成测试用例"

**助手执行步骤**：

1. **下载飞书文档**
   - 使用 `fetch-doc` 技能下载文档
   - 转换为Markdown格式（忽略图片）

2. **分析需求内容**
   - 提取功能模块和业务逻辑
   - 识别测试场景和验证点

3. **生成测试用例文档**
   - 严格按照testcase_template.md格式
   - 确保测试点命名规范
   - 确保验证项符合SMART原则

4. **转换为XMind格式**
   - 执行转换脚本
   - 验证转换结果

5. **提供输出文件**
   - Markdown文档路径
   - XMind思维导图路径
   - 测试统计报告

### 示例3：生成特定模块的测试用例

**用户**："为用户登录模块生成详细的测试用例"

**助手执行步骤**：

1. **理解登录模块需求**
   - 登录方式（账号密码、手机验证码、第三方登录）
   - 登录验证规则
   - 异常处理逻辑

2. **设计测试点**
   - 功能测试：正常登录、各种登录方式
   - 异常测试：错误密码、网络异常、账号锁定
   - 边界测试：密码长度、特殊字符
   - 安全测试：加密传输、防暴力破解

3. **编写验证项**
   - 每个验证项具体可执行
   - 包含前置条件、操作步骤、预期结果

4. **生成文档并转换**
   - 输出符合规范的Markdown和XMind文档

## 注意事项

### 格式要求
- **必须严格遵守** `references/testcase_template.md` 中定义的格式规范
- 所有生成的测试用例文档必须符合标准Markdown格式
- 转换脚本依赖标准格式，格式错误可能导致转换失败

### 质量保障
- 测试用例质量取决于需求文档的清晰度和完整性
- 每个验证项必须具体、可执行、可验证
- 必须包含功能测试、异常测试、边界测试等多种测试类型
- 生成后必须进行质量检查

### 使用限制
- 本技能专注于从需求文档生成测试用例
- 不支持从代码或数据库生成测试用例
- 对于非常大的文档，建议拆分为多个模块分别生成
- 图片内容会被忽略，只处理文字信息

### 输出格式
- 始终生成Markdown和XMind两种格式
- Markdown文档便于版本控制和协作编辑
- XMind思维导图便于可视化展示和评审

### 后续维护
- 需求变更时需要重新生成测试用例
- 建议定期review和更新测试用例
- 可以手动编辑Markdown文档后重新转换为XMind

### 常见问题

**Q: 生成的测试用例格式不正确怎么办？**
A: 请确保SKILL.md中的格式规范与testcase_template.md保持一致，重新生成即可。

**Q: 转换XMind失败怎么办？**
A: 检查Markdown文档是否符合格式规范，特别是标题层级和测试点格式。

**Q: 测试用例覆盖不全怎么办？**
A: 提供更详细的需求文档，或在生成后手动补充遗漏的测试场景。

**Q: 验证项不够具体怎么办？**
A: 在需求文档中提供更多细节，或手动编辑Markdown文档细化验证项。

**Q: 为什么生成的文件名有时间戳后缀？**
A: 为了便于版本管理和历史追溯，每次生成都会自动添加时间戳后缀（格式：-vMMdd_HHmmss）。这样可以保留所有历史版本，便于对比不同版本的测试用例。

**Q: 多个版本的测试用例如何管理？**
A: 建议定期清理不需要的历史版本，只保留最新版本和重要的历史版本。通过文件名中的时间戳可以清晰识别生成时间，便于版本管理。
